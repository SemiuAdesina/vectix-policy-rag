# Pharmacy Inventory Case Studies: Expired Drug Handling and RFC Module

**Policy ID:** VL-OPS-018-CS  
**Effective Date:** January 1, 2026  
**Focus:** Historical examples, audit practices, and troubleshooting for expired drug management and the Return for Credit (RFC) module.  
**Version:** 2.0.1

---

## 1. Introduction and Context

VectixLogic’s pharmacy inventory system supports daily stock management, expiry tracking, and the Return for Credit (RFC) workflow for eligible expired or soon-to-expire products. This document provides real-world case studies from 2024–2025, sample audit log patterns, and step-by-step troubleshooting for the RFC module so that pharmacists and operations staff can resolve common issues without escalating to engineering.

---

## 2. Case Study 1: Batch Expiry at UCH (Q2 2024)

### 2.1 Situation

A batch of Schedule II controlled substances was approaching expiry within 30 days. The system had correctly flagged the batch in the “Expiring Soon” report, but the physical count performed on a Friday did not match the system quantity for one strength of the product. The discrepancy was discovered during the mandatory weekly audit.

### 2.2 Root Cause

A partial dispense had been recorded in the system without a corresponding update to the batch-level quantity due to a user error (wrong batch selected at dispense). The system’s “Expiring Soon” list was driven by batch-level records, so the count mismatch was a data integrity issue rather than a software defect.

### 2.3 Resolution

- The pharmacy team performed a full physical count for all batches of that product and reconciled each batch in the system.
- The incorrect dispense was corrected by moving the dispensed quantity from the wrong batch to the correct batch (with a documented correction log).
- Going forward, the site adopted the practice of verifying the batch selected at dispense against the physical bottle barcode before confirming.

### 2.4 Takeaway

When system and physical counts disagree, first reconcile at the batch level. Use the audit log (see Section 4) to trace dispenses and adjustments by batch ID and user. Correct data entry errors via documented adjustments rather than ad-hoc overrides.

## 3. Case Study 2: Return for Credit (RFC) Recovery Report (2024–2025)

### 3.1 Background

The RFC module allows sites to submit eligible expired or short-dated inventory to manufacturers (or their designated reverse-logistics partners) for credit. Eligibility is determined by the manufacturer’s policy and the product’s inclusion in our RFC eligibility list, which is maintained by VectixLogic in consultation with the site and the manufacturer.

### 3.2 Example Recovery Summary (Fiscal 2024–2025)

The following is an anonymized summary of RFC activity for a 12-month period at a single site using VectixLogic software:

| Metric | Value |
|--------|--------|
| Total batches moved to Quarantine Zone (expired) | 47 |
| Batches submitted for RFC | 38 |
| Batches ineligible for RFC (e.g., manufacturer not in program, product excluded) | 9 |
| RFC forms generated by system | 38 |
| Credits received (value) | As per site finance records |
| Average time from quarantine to RFC form generation | 3 business days |
| Disputes or rejections by manufacturer | 2 (resolved with supporting documentation) |

### 3.3 Observations

- Two manufacturer rejections were due to incomplete documentation (missing destruction certificate for one batch; incorrect NAFDAC number on the form for another). Both were corrected and resubmitted successfully.
- Sites that perform weekly “Expiring Soon” reviews and initiate RFC early (before actual expiry when allowed by manufacturer) tend to have higher approval rates and faster credit turnaround.

## 4. Audit Log Conventions and How to Use Them

### 4.1 What the System Logs

The pharmacy inventory system writes immutable audit records for:

- **Stock movements:** Receipts, dispenses, transfers, adjustments, and moves to/from Quarantine Zone.
- **User actions:** User ID, timestamp, action type, and relevant identifiers (batch ID, product ID, quantity, reason code if applicable).
- **RFC lifecycle:** RFC form creation, submission, and status updates (e.g., “Submitted,” “Credit Issued,” “Rejected”).

### 4.2 Sample Audit Log Entries (Conceptual)

**Dispense:**
```
2024-06-14T14:32:01Z | USER: pharmacist_uch_01 | ACTION: DISPENSE | PRODUCT_ID: P-xxx | BATCH_ID: B-yyy | QTY: -10 | BALANCE_AFTER: 90
```

**Move to Quarantine:**
```
2024-06-20T09:00:00Z | USER: pharmacist_uch_01 | ACTION: MOVE_TO_QUARANTINE | BATCH_ID: B-zzz | REASON: EXPIRED | WITNESS: pharmacist_uch_02
```

**RFC form generated:**
```
2024-06-23T11:15:00Z | USER: inventory_clerk_01 | ACTION: RFC_FORM_GENERATED | FORM_ID: RFC-2024-0678 | BATCH_IDS: [B-zzz, ...] | STATUS: PENDING_SUBMISSION
```

### 4.3 Using Logs for Reconciliation and Troubleshooting

- **Count discrepancy:** Filter by `BATCH_ID` and `ACTION` (e.g., DISPENSE, ADJUSTMENT) for the date range in question; compare to physical movement records.
- **RFC status:** Search by `FORM_ID` or `BATCH_ID` to see the full lifecycle of an RFC submission.
- **Compliance:** Export logs for a given period for PCN or FDA audit; ensure retention meets the 5-year requirement per VL-COMP-011.

## 5. Troubleshooting the Return for Credit (RFC) Module

### 5.1 RFC Form Fails to Generate

**Symptom:** User clicks “Generate RFC” for selected quarantined batches but no form is produced, or an error message appears.

**Checks:**
1. **Eligibility:** Confirm each selected batch is on the RFC eligibility list (product and manufacturer). Ineligible batches must be removed from the selection or handled separately (e.g., destruction only).
2. **Required fields:** Ensure each batch has a valid NAFDAC number (Nigeria) or equivalent identifier required by the manufacturer. Missing or invalid identifiers will cause the form to fail validation.
3. **Permissions:** The generating user must have the “RFC: Create” permission. If the button is disabled or an “Access denied” message appears, contact the site administrator to assign the correct role.
4. **System status:** Verify that the RFC service is not in maintenance mode (check the status page or with IT). If the issue persists, export the list of affected batch IDs and contact VectixLogic support with the export and a screenshot of the error.

### 5.2 RFC Form Submitted but No Credit Received

**Symptom:** Form shows “Submitted” but the manufacturer has not issued credit within the expected window (e.g., 30–60 days per manufacturer policy).

**Checks:**
1. **Tracking:** Confirm with the manufacturer or their logistics partner that the physical return was received and matched to the form. Mismatched quantities or damaged packaging can delay processing.
2. **Documentation:** Ensure the destruction certificate (for any portion destroyed on-site, if applicable) and the signed packing list were included as required by the manufacturer. Resubmit or send supplementary documents if requested.
3. **Dispute:** If the manufacturer rejects the claim, the system will typically show a “Rejected” status and a reason code. Address the reason (e.g., correct NAFDAC number, add missing batch details) and resubmit or follow the manufacturer’s appeal process.

### 5.3 Batch Stuck in “Quarantine” or “Pending RFC”

**Symptom:** A batch appears in Quarantine or “Pending RFC” but cannot be selected for RFC or moved to destruction.

**Checks:**
1. **Duplicate or locked:** Another user may have the batch in an open RFC draft. Ensure no other user has an unsaved or in-progress RFC that includes this batch. If necessary, an admin can release the lock (per your site’s procedure).
2. **Witness requirement:** For controlled substances, destruction or RFC may require a second licensed pharmacist to witness. Ensure the “Witness” field is completed in the audit log for the move to quarantine; if the workflow requires witness confirmation before RFC, that step must be completed.
3. **Data fix:** If the batch is in an inconsistent state (e.g., quantity zero but status still “Quarantine”), do not attempt to fix data manually. Log the batch ID and contact VectixLogic support for a data repair; document the incident for your own audit trail.

### 5.4 Export or Report Shows Wrong Totals

**Symptom:** The RFC recovery report or an exported list does not match the expected number of batches or total value.

**Checks:**
1. **Date range and filters:** Confirm the date range and any filters (site, product, status) match what you intend. “Submitted” vs. “Credit Issued” will change counts.
2. **Currency and rounding:** Value reports may be in a single currency and rounded; check the report footer for currency and rounding rules.
3. **Cached report:** If the report is cached, refresh or regenerate it. If the discrepancy remains, export the underlying audit log entries for the same period and reconcile manually; then report the discrepancy to support so we can fix any reporting bug.

## 6. Best Practices Summary

- **Weekly audits:** Perform physical counts and system reconciliation for Schedule II and other high-value or high-risk products every week, and resolve discrepancies before moving batches to quarantine or RFC.
- **Early RFC:** Review “Expiring Soon” lists regularly and initiate RFC as soon as batches are eligible to reduce write-offs and speed up credit.
- **Documentation:** Keep destruction certificates and RFC submission confirmations in a single place (e.g., shared drive or document management system) and reference them in the system where the workflow allows.
- **Audit logs:** Use audit logs as the first place to look when investigating discrepancies or RFC status; only escalate to engineering when the logs indicate a system error or data inconsistency that cannot be corrected at the site.

## 8. Appendix: Definitions

| Term | Definition |
|------|------------|
| **Audit log** | Immutable record of user actions and system events (e.g., dispense, move to quarantine) used for reconciliation and compliance. |
| **Batch** | A distinct quantity of a product received under a single lot and expiry; the unit of tracking for inventory and RFC. |
| **Batch ID** | Unique identifier assigned to a batch in the system (e.g., B-yyy). |
| **Controlled substances** | Drugs regulated under Schedule II or equivalent; require weekly physical count and witness for destruction per VL-OPS-003. |
| **Credit Issued** | RFC status indicating the manufacturer has approved and processed the return for credit. |
| **Destruction certificate** | Document attesting to the lawful destruction of expired or illegitimate product; required by some manufacturers for RFC. |
| **Dispense** | Recording the release of product to a patient or another location; reduces on-hand quantity and is logged in the audit trail. |
| **Eligibility list** | List of products and manufacturers approved for Return for Credit; maintained by VectixLogic in consultation with sites. |
| **Expiring Soon** | Report or list of batches expiring within a defined window (e.g., 30 days); used to trigger RFC or disposal. |
| **FORM_ID** | Unique identifier for an RFC form (e.g., RFC-2024-0678). |
| **NAFDAC** | National Agency for Food and Drug Administration and Control (Nigeria); product registration and identifiers used in RFC. |
| **Quarantine Zone** | System state for product that is expired or suspect; product cannot be dispensed until cleared or destroyed. |
| **Return for Credit (RFC)** | Process by which eligible expired or short-dated inventory is returned to the manufacturer for credit. |
| **Witness** | Second licensed pharmacist required to witness destruction or certain moves to quarantine for controlled substances. |
| **PENDING_SUBMISSION** | RFC form status indicating the form has been generated but not yet sent to the manufacturer. |
